## 0. 시작하기 전에 — 왜 계정을 나눠?

### 🤔 한 사람이 다 root로 쓰면 안 돼?

기술적으론 가능하지만, **현실에선 절대 그러면 안 돼요**. 이유:

#### ❌ 다 root로 쓸 때의 문제

- 🐛 **실수 한 번에 전체 시스템 날아감** — `rm -rf /` 한 줄로 끝
- 🕵️ **누가 뭘 했는지 추적 불가** — 모든 로그가 "root가 했음"
- 🚪 **해킹당하면 끝장** — 공격자가 root 권한 통째로 얻음
- 👥 **협업 시 충돌** — 다른 사람 파일 모르고 덮어쓰기 가능

#### ✅ 역할별로 계정 나누면

- 🛡️ **실수해도 본인 영역만 영향** — 시스템 보호
- 📝 **누가 뭘 했는지 로그에 남음** — 책임 추적 가능
- 🔒 **해킹당해도 피해 제한적** — 권한 분리
- 👥 **협업 시 충돌 방지** — 권한으로 영역 구분

### 🤔 이번 미션에서 만들 계정 3개

| 계정 | 역할 | 비유 |
|---|---|---|
| **agent-admin** | 운영/관리, cron 실행 | 시스템 관리자 (정해진 일을 자동으로 시키는 사람) |
| **agent-dev** | 개발/운영, 스크립트 작성 | 개발자 (코드 짜고 배포하는 사람) |
| **agent-test** | QA/테스트 | 테스터 (만든 거 확인하는 사람) |

### 🤔 그룹은 또 왜 만들어?

**여러 사람에게 같은 권한을 한 번에 주려고**요. 

비유: 회사에서 사원증 권한을 한 명씩 따로 주는 게 아니라, "개발팀", "디자인팀" 부서별로 부여하잖아요. 같은 개념.

| 그룹 | 멤버 | 무슨 일 하나? |
|---|---|---|
| **agent-common** | admin, dev, test | 공유 폴더 (upload_files) 접근용 |
| **agent-core** | admin, dev | 보안 폴더 (api_keys, 로그) 접근용 |

> 핵심: **test 계정은 보안 폴더 접근 불가**. 테스터한테 API 키 보여줄 필요 없잖아요.

## 1. 핵심 용어 미리 알기

| 용어 | 비유 | 진짜 의미 |
|---|---|---|
| **User** | 사람 한 명 | 시스템 사용자 계정 (사람 또는 프로그램용) |
| **UID** | 사람의 주민번호 | User ID, 사용자 식별 번호 |
| **Group** | 부서/팀 | 사용자들의 묶음 |
| **GID** | 부서 코드 | Group ID, 그룹 식별 번호 |
| **Primary Group** | 본부서 (소속) | 사용자가 만든 파일의 기본 그룹 |
| **Secondary Group** | 겸직 부서 | 추가로 속한 그룹들 |
| **/home/사용자명** | 그 사람의 개인 사무실 | 사용자의 홈 디렉토리 |
| **chmod** | 출입증 권한 변경 | change mode (권한 변경) |
| **chown** | 사무실 주인 변경 | change owner (소유자 변경) |
| **chgrp** | 사무실 담당 부서 변경 | change group (그룹 변경) |
| **ACL** | 출입증 외 추가 카드키 | Access Control List (세밀한 권한) |

## 2. Linux 권한의 기본 — rwx 읽는 법

### 🔍 `ls -l` 결과 해석

```bash
ls -l /home/agent-admin/agent-app
```

이런 결과가 나와요:
```text
drwxr-xr-x  4 agent-admin agent-common 4096 May 18 10:00 upload_files
-rw-r-----  1 agent-admin agent-core    20 May 18 10:00 t_secret.key
```

### 🔍 첫 줄(`drwxr-xr-x`)을 잘게 쪼개면

```text
d   rwx    r-x    r-x
│    │      │      │
│    │      │      └── 그 외 모두(others)의 권한
│    │      └────────  그룹(group)의 권한
│    └───────────────  소유자(owner)의 권한
└──── 파일 종류 (d=디렉토리, -=일반파일, l=링크)
```

### 🔤 r, w, x 각각의 의미

| 글자 | 파일에서는 | 디렉토리에서는 |
|---|---|---|
| `r` (read) | 내용 읽기 | 목록 보기 (`ls`) |
| `w` (write) | 내용 수정 | 파일 생성/삭제 |
| `x` (execute) | 실행 가능 | 안으로 들어가기 (`cd`) |
| `-` | 권한 없음 | 권한 없음 |

> 💡 **혼동 주의**: 디렉토리에서 `x` 권한이 없으면 안에 *들어갈 수도 없어요*. `r`만 있고 `x` 없으면 목록은 보이는데 그 안에 못 들어감.

### 🔢 숫자로 표현하기 (chmod 750의 의미)

각 자리 (r, w, x)를 숫자로:

| 권한 | 숫자 |
|---|---|
| `r` | 4 |
| `w` | 2 |
| `x` | 1 |
| `-` | 0 |

조합해서 더하면:
- `rwx` = 4+2+1 = **7**
- `rw-` = 4+2 = **6**
- `r-x` = 4+1 = **5**
- `r--` = 4 = **4**
- `---` = 0 = **0**

그래서 `chmod 750` = **`rwxr-x---`** (소유자 전부, 그룹은 읽기/실행, 나머진 없음)

이번 미션의 핵심 권한:

| 권한 | 숫자 | 누가 뭘 할 수 있나 |
|---|---|---|
| `rwxr-x---` | 750 | 소유자: 전부 / 그룹: 읽기+들어가기 / 나머지: 아무것도 |
| `rwxrwx---` | 770 | 소유자, 그룹: 전부 / 나머지: 아무것도 |
| `rw-r-----` | 640 | 소유자: 읽기+쓰기 / 그룹: 읽기만 / 나머지: 아무것도 |

## 3. ACL이 뭐고 왜 필요해?

### 🤔 기본 rwx의 한계

기본 권한은 **딱 3종류 (소유자/그룹/그외)** 만 설정 가능해요. 근데 현실은 더 복잡해요:

> "이 폴더는 dev 그룹은 읽기/쓰기, test 그룹은 읽기만, 그 외엔 차단"

이런 거 기본 rwx로 못 해요. 그룹은 하나만 설정 가능하니까.

### ✅ ACL이 해주는 것

ACL(Access Control List) = **여러 사용자/그룹에 *각자 다른* 권한을 주는 시스템**

비유:
- 기본 권한 = 사무실 문 카드키 한 종류만 가능 (모든 직원 같은 등급)
- ACL = 사람마다, 부서마다 카드키 등급 다르게 가능

### 🔍 ACL 명령어 두 개

| 명령 | 하는 일 |
|---|---|
| `setfacl` | ACL 설정 (set) |
| `getfacl` | ACL 확인 (get) |

### 💡 이번 미션에서 ACL이 필요한 이유

요구사항을 다시 보면:
- `upload_files/` → agent-common 그룹 R/W
- `api_keys/`, `/var/log/agent-app/` → **agent-core 그룹만** R/W

근데 디렉토리의 그룹은 *하나*만 가능. 그래서 기본 그룹은 agent-core로 하고, 만약 추가로 다른 사용자/그룹에도 권한 줘야 한다면 → **ACL로 추가**하면 돼요. 

이번 미션 자체는 그룹 하나로도 처리 가능한 구조지만, **ACL 검증 명령어 (`getfacl`)** 가 제출 요건에 들어있으니까 익혀두면 좋아요.

## 4. 실습 단계

### Step 1. 사용자 3명 만들기

#### 🎯 목적
agent-admin, agent-dev, agent-test 계정 생성.

#### ⌨️ 명령어
```bash
# 패키지 설치 (sudo, acl 미리 깔아두기)
apt update
apt install -y sudo acl

# 사용자 3명 생성 (홈 디렉토리 자동 생성, bash 셸 사용)
useradd -m -s /bin/bash agent-admin
useradd -m -s /bin/bash agent-dev
useradd -m -s /bin/bash agent-test

# 비밀번호 설정 (각각 입력해야 함)
passwd agent-admin
passwd agent-dev
passwd agent-test
```

#### 🔍 옵션 뜯어보기
| 옵션 | 의미 |
|---|---|
| `-m` | **m**ake home directory (홈 폴더 자동 생성) |
| `-s /bin/bash` | 로그인 시 사용할 **s**hell 지정 |

#### 🤔 `useradd` vs `adduser` 차이?
| 명령 | 특징 |
|---|---|
| `useradd` | 저수준. 옵션 직접 지정해야 함 |
| `adduser` | 고수준. 대화형으로 비번/이름 등 물어봄 |

둘 다 가능. 이번엔 `useradd`로 옵션 명시적으로.

#### ✅ 확인
```bash
cat /etc/passwd | grep agent
```

결과 예시:
```text
agent-admin:x:1000:1000::/home/agent-admin:/bin/bash
agent-dev:x:1001:1001::/home/agent-dev:/bin/bash
agent-test:x:1002:1002::/home/agent-test:/bin/bash
```

각 필드 의미: `사용자명:비번(x=암호화됨):UID:GID::홈디렉토리:기본셸`

### Step 3. 사용자를 그룹에 추가

#### 🎯 목적
요구사항대로 멤버 배치:
- **agent-common** = admin + dev + test
- **agent-core** = admin + dev

#### ⌨️ 명령어
```bash
# agent-common 그룹에 3명 모두 추가
usermod -aG agent-common agent-admin
usermod -aG agent-common agent-dev
usermod -aG agent-common agent-test

# agent-core 그룹에 admin, dev 추가 (test 제외!)
usermod -aG agent-core agent-admin
usermod -aG agent-core agent-dev
```

#### 🔍 옵션 뜯어보기
| 옵션 | 의미 |
|---|---|
| `usermod` | **user mod**ify (사용자 정보 수정) |
| `-a` | **a**ppend (덧붙이기) |
| `-G` | secondary **G**roup (추가 그룹) |

#### ⚠️ 진짜 자주 하는 실수
**`-a` 없이 `-G`만 쓰면 기존 그룹이 다 날아가요!**
```bash
# ❌ 위험! 기존 그룹 다 사라짐
usermod -G agent-common agent-admin

# ✅ 안전. 기존 그룹 유지하면서 추가
usermod -aG agent-common agent-admin
```

**`-aG`는 무조건 같이 쓴다**고 외우세요.

#### ✅ 확인
```bash
id agent-admin
id agent-dev
id agent-test
```

결과 예시:
```text
uid=1000(agent-admin) gid=1000(agent-admin) groups=1000(agent-admin),1003(agent-common),1004(agent-core)
uid=1001(agent-dev)   gid=1001(agent-dev)   groups=1001(agent-dev),1003(agent-common),1004(agent-core)
uid=1002(agent-test)  gid=1002(agent-test)  groups=1002(agent-test),1003(agent-common)
```

핵심 체크:
- ✅ agent-admin과 agent-dev → `agent-common` + `agent-core` 둘 다 보임
- ✅ agent-test → `agent-common`만 보임 (agent-core 없음!)

### Step 4. 디렉토리 구조 만들기

#### 🎯 목적
요구사항에 맞춰 디렉토리 생성.

#### ⌨️ 명령어
```bash
# agent-admin 홈 아래에 agent-app 디렉토리 생성 (이게 AGENT_HOME)
mkdir -p /home/agent-admin/agent-app/upload_files
mkdir -p /home/agent-admin/agent-app/api_keys
mkdir -p /home/agent-admin/agent-app/bin

# 로그 디렉토리 생성
mkdir -p /var/log/agent-app
```

#### 🔍 뜯어보기
- `mkdir` = make directory
- `-p` = parents (중간 폴더도 자동 생성, 이미 있어도 에러 안 남)

#### 📁 결과적으로 만든 구조
```text
/home/agent-admin/agent-app/        ← $AGENT_HOME
├── upload_files/                   ← 공유 폴더 (agent-common)
├── api_keys/                       ← 보안 폴더 (agent-core)
└── bin/                            ← 스크립트 보관 (monitor.sh 등)

/var/log/agent-app/                 ← 로그 폴더 (agent-core)
```

#### ✅ 확인
```bash
ls -la /home/agent-admin/agent-app/
ls -la /var/log/agent-app
```

### Step 5. 소유자와 그룹 지정 (chown)

#### 🎯 목적
디렉토리마다 *올바른 그룹*을 매칭.

#### ⌨️ 명령어
```bash
# AGENT_HOME 자체와 그 안 디렉토리들의 소유자를 agent-admin으로
chown -R agent-admin:agent-admin /home/agent-admin/agent-app

# upload_files → 그룹을 agent-common으로
chown agent-admin:agent-common /home/agent-admin/agent-app/upload_files

# api_keys → 그룹을 agent-core로
chown agent-admin:agent-core /home/agent-admin/agent-app/api_keys

# /var/log/agent-app → 그룹을 agent-core로
chown agent-admin:agent-core /var/log/agent-app
```

#### 🔍 뜯어보기
- `chown` = **ch**ange **own**er
- `사용자:그룹` 형식으로 소유자, 그룹 동시 변경
- `-R` = **R**ecursive (하위까지 다 적용)

#### 🤔 왜 첫 줄에서 `-R`을 쓰고 그 다음엔 안 써?
- 첫 줄 = "전체를 일단 admin:admin으로 통일"
- 그 다음 = "특정 디렉토리만 그룹 바꿈" (재귀로 하면 안 됨, 안의 파일은 admin:admin 유지하고 디렉토리 그룹만 바꾸려는 거)

#### ✅ 확인
```bash
ls -la /home/agent-admin/agent-app/
```

결과 기대값:
```text
drwxr-xr-x 4 agent-admin agent-admin   ...  .
drwxr-xr-x 2 agent-admin agent-common  ...  upload_files
drwxr-xr-x 2 agent-admin agent-core    ...  api_keys
drwxr-xr-x 2 agent-admin agent-admin   ...  bin
```

### Step 6. 권한 설정 (chmod)

#### 🎯 목적
각 디렉토리에 *딱 맞는 권한* 부여.

#### ⌨️ 명령어
```bash
# AGENT_HOME: 소유자 전부, 그룹 읽기+실행, 나머지 차단
chmod 750 /home/agent-admin/agent-app

# upload_files: 그룹도 R/W 가능 (770)
chmod 770 /home/agent-admin/agent-app/upload_files

# api_keys: 그룹만 R/W, 나머진 완전 차단 (770)
chmod 770 /home/agent-admin/agent-app/api_keys

# bin: 소유자/그룹 R/W/X, 나머지 차단 (750)
chmod 750 /home/agent-admin/agent-app/bin

# 로그 디렉토리: 그룹 R/W, 나머지 차단 (770)
chmod 770 /var/log/agent-app
```

#### 🔍 권한 의미 요약

| 디렉토리 | 권한 | 의미 |
|---|---|---|
| `agent-app` | 750 | agent-admin 전부 / agent-common 그룹 진입 가능 / 나머지 차단 |
| `upload_files` | 770 | agent-admin 전부 / agent-common 그룹 R/W / 나머지 차단 |
| `api_keys` | 770 | agent-admin 전부 / **agent-core 그룹**만 R/W / 나머지 차단 |
| `bin` | 750 | 스크립트 보관용, 일반 진입 가능 |
| `/var/log/agent-app` | 770 | agent-admin 전부 / **agent-core 그룹**만 R/W |

#### 🚨 왜 770이지 777이 아닐까?
- `777` = 누구나 다 가능 → **절대 금물**. 모르는 사람도 접근 가능
- `770` = 소유자/그룹만 가능, 나머진 차단 → **최소 권한 원칙**

> **최소 권한 원칙(Principle of Least Privilege)**: 필요한 만큼만 권한 줌. 보안 기본 중의 기본.

#### ✅ 확인
```bash
ls -la /home/agent-admin/agent-app/
```

`rwxrwx---`(770) 또는 `rwxr-x---`(750) 형태로 나오면 OK.

### Step 7. ACL로 더 정교한 권한 추가 (선택 + 검증용)

#### 🎯 목적
기본 권한 외에 ACL로 추가 권한 설정 + `getfacl` 결과 확보 (제출용).

#### ⌨️ 명령어 (예시 — 요구사항 보강용)
```bash
# upload_files: agent-common 그룹이 새 파일을 만들어도 자동으로 그룹 권한 유지
setfacl -d -m g:agent-common:rwx /home/agent-admin/agent-app/upload_files

# api_keys: 디폴트 ACL로 agent-core 그룹 권한 자동 상속
setfacl -d -m g:agent-core:rwx /home/agent-admin/agent-app/api_keys
setfacl -d -m g:agent-core:rwx /var/log/agent-app
```

#### 🔍 옵션 뜯어보기
| 옵션 | 의미 |
|---|---|
| `setfacl` | ACL 설정 |
| `-d` | **d**efault (이 폴더에 새로 만들어지는 파일에도 자동 적용) |
| `-m` | **m**odify (수정/추가) |
| `g:agent-common:rwx` | **g**roup `agent-common`에 `rwx` 부여 |

#### 🤔 `-d`(default)가 왜 중요해?
ACL 없으면, 폴더에 새 파일 만들 때 *기본 권한*만 적용돼요. 그래서 그룹 멤버가 만든 파일을 다른 그룹 멤버가 못 읽는 일이 생김.

`-d` ACL은 **"이 폴더에 만들어지는 모든 새 파일에 자동으로 이 권한을 적용해줘"** 라는 뜻. 협업 디렉토리에 특히 중요.

#### ✅ 확인
```bash
getfacl /home/agent-admin/agent-app/upload_files
getfacl /home/agent-admin/agent-app/api_keys
getfacl /var/log/agent-app
```

결과 예시:
```text
# file: upload_files
# owner: agent-admin
# group: agent-common
user::rwx
group::rwx
other::---
default:user::rwx
default:group::rwx
default:group:agent-common:rwx
default:mask::rwx
default:other::---
```

`default:` 줄들이 *자동 상속될 권한*이에요.

### Step 8. agent-admin 홈 디렉토리 접근권 점검

#### 🎯 목적
다른 사용자(예: agent-dev)가 agent-admin의 `agent-app` 폴더 안에 들어갈 수 있어야 함. 근데 `/home/agent-admin` 자체에 진입 권한이 없으면 안 됨.

#### ⌨️ 명령어
```bash
# /home/agent-admin 폴더에 대해 다른 그룹원도 진입(x)할 수 있게
chmod 755 /home/agent-admin
```

#### 🔍 왜 이게 필요해?
경로를 따라가려면 *각 단계마다* `x` 권한이 필요해요.

- `/home/agent-admin/agent-app/upload_files`로 진입하려면
  - `/` (x 필요) ✅
  - `/home` (x 필요) ✅
  - `/home/agent-admin` (x 필요) ← **이게 차단되면 못 들어감**
  - `/home/agent-admin/agent-app` (x 필요)
  - `/home/agent-admin/agent-app/upload_files` (x 필요)

#### ⚠️ 트레이드오프
`/home/agent-admin`을 755로 열면 *다른 사용자가 이 폴더 안 목록을 볼 수 있음*. 단, 안의 파일/폴더에 권한이 없으면 내용은 못 봐요. 

더 엄격하게 하려면 711(목록도 못 보게)도 가능: `chmod 711 /home/agent-admin`

### Step 9. 사용자 전환 테스트

#### 🎯 목적
실제로 권한이 잘 분리됐는지 *테스트*.

#### ⌨️ 명령어
```bash
# agent-test로 전환
su - agent-test

# 들어왔으면 테스트
whoami                                                  # → agent-test
cd /home/agent-admin/agent-app/upload_files             # 진입 가능해야 함
touch test.txt                                          # 파일 만들어보기 (성공해야 함)

cd /home/agent-admin/agent-app/api_keys                 # → Permission denied 떠야 정상!
cat /var/log/agent-app/anything                         # → Permission denied 떠야 정상!

# 나가기
exit
```

#### 🔍 `su -` 의미
- `su` = **s**witch **u**ser
- `-` (대시) = 새 셸 환경을 *깨끗하게* 시작 (환경변수 새로 로드)

#### ✅ 기대 결과

| 동작 | agent-admin | agent-dev | agent-test |
|---|---|---|---|
| `upload_files`에 파일 생성 | ✅ 가능 | ✅ 가능 | ✅ 가능 |
| `api_keys` 진입 | ✅ 가능 | ✅ 가능 | ❌ **거부** |
| `/var/log/agent-app` 진입 | ✅ 가능 | ✅ 가능 | ❌ **거부** |

agent-test가 보안 디렉토리에서 막히면 → **권한 분리 성공!** 🎉

## 6. 자주 만나는 에러와 해결법

### ❌ `useradd: command not found`
**원인**: 컨테이너 최소 이미지에 일부 명령이 없음
**해결**:
```bash
apt update && apt install -y passwd
```

### ❌ `usermod` 후에 변경이 안 보임
**원인**: 이미 로그인된 세션은 기존 그룹 정보 유지
**해결**: 해당 사용자로 다시 로그인하거나 `newgrp 그룹명`으로 갱신

### ❌ `setfacl: command not found`
**원인**: acl 패키지가 없음
**해결**:
```bash
apt install -y acl
```

### ❌ `mkdir: cannot create directory`
**원인**: 권한 부족 (root가 아닌 사용자)
**해결**: root로 작업하거나 `sudo` 사용

### ❌ 사용자 전환했는데 명령어 일부가 안 됨
**원인**: PATH 환경변수 차이
**해결**: `su - 사용자명`처럼 대시(`-`) 붙여서 환경 새로 로드

### ❌ 다른 그룹 멤버가 만든 파일에 접근 안 됨
**원인**: 새 파일은 *만든 사람의 기본 그룹*으로 생김
**해결 1**: 디폴트 ACL 설정 (`setfacl -d -m ...`)
**해결 2**: 디렉토리에 SGID 비트 설정 (`chmod g+s 폴더명`)

#### 💡 SGID 비트란?
디렉토리에 `g+s`를 주면, 그 안에 만들어지는 파일이 *디렉토리의 그룹*을 자동으로 상속받음. 협업 디렉토리에 매우 유용.

```bash
chmod g+s /home/agent-admin/agent-app/upload_files
```

확인 시 그룹 자리에 `s`가 보임: `rwxrws---`

## 7. 전체 흐름 한눈에 보기

```text
[ root 권한으로 시작 ]
     │
     │ apt install -y sudo acl                          ← 도구 준비
     │
     │ useradd -m -s /bin/bash agent-admin              ← 사용자 3명 생성
     │ useradd -m -s /bin/bash agent-dev
     │ useradd -m -s /bin/bash agent-test
     │ passwd 각각                                       ← 비밀번호 설정
     │
     │ groupadd agent-common                            ← 그룹 2개 생성
     │ groupadd agent-core
     │
     │ usermod -aG agent-common agent-admin             ← 그룹 멤버 배치
     │ usermod -aG agent-common agent-dev               
     │ usermod -aG agent-common agent-test              
     │ usermod -aG agent-core   agent-admin             
     │ usermod -aG agent-core   agent-dev               ← (test는 안 함!)
     │
     │ mkdir -p /home/agent-admin/agent-app/{upload_files,api_keys,bin}
     │ mkdir -p /var/log/agent-app                      ← 디렉토리 구조 생성
     │
     │ chown 각각 적절한 그룹                            ← 소유권 지정
     │ chmod 770/750 적절히                             ← 권한 설정
     │
     │ setfacl -d -m g:그룹:rwx 디렉토리                 ← 디폴트 ACL (선택)
     │ chmod 755 /home/agent-admin                      ← 경로 진입 보장
     │
     │ su - agent-test → 권한 분리 테스트               ← 검증
     ▼
[ 완료: 협업 + 보안 분리 완료 ]
```

## 📌 마무리: 이번에 진짜 배운 것

1. **"역할로 나눠라"** — admin/dev/test, 사람마다 할 일과 권한이 다름
2. **"그룹으로 권한을 묶어라"** — 개별 사용자 권한 설정은 관리 지옥
3. **"최소 권한 원칙"** — 필요한 만큼만 주기. 777은 거의 항상 답이 아님
4. **"공유 ≠ 모두 다"** — 공유 디렉토리도 *지정된 그룹*만 접근
5. **"ACL은 강력하다"** — 기본 rwx로 부족할 때 ACL이 답
6. **"테스트로 검증한다"** — 권한 설정 후 실제 사용자로 전환해 확인
