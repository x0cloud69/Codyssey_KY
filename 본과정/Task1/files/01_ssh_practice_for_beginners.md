## 0. 시작하기 전에 — 큰 그림 먼저

### 🤔 우리가 지금 뭐 하려는 거야?

한 문장으로: **"내 Mac 안에 가짜 Linux 서버를 하나 만들고, 그 서버에 원격 접속하는 연습"** 이에요.

### 🤔 그게 왜 필요해?

회사 서버나 클라우드 서버는 보통 **저 멀리 어딘가**에 있어요 (AWS, GCP, 회사 데이터센터 등). 그런 서버를 만질 때는 **SSH** 라는 기술로 원격 접속해서 명령어로 다루는 게 표준이에요.

근데 처음부터 진짜 서버로 연습하면:
- 💸 돈이 들고
- 💥 망가뜨릴까봐 무섭고
- 🐢 설정이 복잡해서 시작도 못 함

그래서 → **내 Mac 안에 "가상 Linux 서버"를 만들어서 마음껏 망가뜨려도 되는 연습장**을 만드는 거예요. 망가지면 컨테이너 지우고 다시 만들면 끝!

### 🤔 그래서 결과적으로 뭘 배우는 건데?

- ✅ Docker로 가짜 Linux 만드는 법
- ✅ Linux 안에서 프로그램 설치하는 법 (`apt`)
- ✅ 설정 파일 수정하는 법 (`vim`)
- ✅ SSH 서버 켜고/끄고/재시작하는 법
- ✅ 보안 기본 설정 (포트 바꾸기, root 차단)

## 1. 핵심 용어 미리 알기

명령어 들어가기 전에, 자주 나올 단어들을 **일상 비유**로 먼저 알아둘게요.

| 용어 | 비유 | 진짜 의미 |
|---|---|---|
| **Mac (호스트)** | 내가 살고 있는 큰 집 | 우리가 쓰는 컴퓨터 본체 |
| **OrbStack** | 집 안에 작은 별채 짓는 도구 | Mac에서 Docker/Linux를 쉽게 돌리는 앱 |
| **Docker** | 별채 안에 들어가는 캡슐 호텔 | 가상 환경 격리 기술 |
| **컨테이너** | 캡슐 호텔의 방 하나 | Docker가 만든 격리된 실행 공간 |
| **Ubuntu** | 그 방 안에 들어 있는 가구 세트 | Linux 운영체제의 한 종류 |
| **SSH** | 외부에서 그 방에 들어가는 *암호화된* 전용 통로 | 원격 접속 + 암호화 통신 프로토콜 |
| **포트(Port)** | 그 방의 *문 번호* | 컴퓨터 통신 출입구 번호 (0~65535) |
| **root** | 마스터키를 가진 사람 | 모든 권한을 가진 최고 관리자 계정 |
| **sshd** | 그 방에서 SSH 통로를 열어두고 손님을 기다리는 직원 | SSH 서버 프로그램 (d = daemon = 백그라운드 직원) |
| **apt** | Ubuntu의 앱스토어 | Ubuntu 패키지(프로그램) 설치/관리 도구 |
| **vim** | 메모장의 *마니아용* 버전 | Linux 터미널 텍스트 편집기 |

💡 **꼭 기억할 한 가지**: `sshd` 끝에 붙은 **`d`** 는 **daemon(데몬)** 의 약자예요. 데몬은 "백그라운드에서 조용히 일하는 프로그램"이라는 뜻. Linux에서 `~d`로 끝나는 건 거의 다 백그라운드 서비스라고 보면 돼요.

## 2. 전체 구조 이해 (집 모양으로)

```text
🏠 Mac (macOS)                          ← 진짜 내 컴퓨터
  │
  └─ 🏗️ OrbStack                        ← Docker를 쉽게 돌려주는 도구
       │
       └─ 📦 Docker Container           ← 격리된 가상 공간 한 칸
            │
            └─ 🐧 Ubuntu Linux          ← 그 안에서 돌아가는 OS
                 │
                 └─ 🔐 OpenSSH Server   ← 외부 접속을 받아주는 프로그램
```

**읽는 법**: 위에서 아래로 "안에 들어 있다"는 뜻이에요.
> Mac 안에 OrbStack이 있고, 그 안에 Docker 컨테이너가 있고, 그 안에 Ubuntu가 돌고 있고, Ubuntu 안에 SSH 서버가 켜져 있다.

이렇게 만들면 **"바깥(Mac)에서 안쪽(Ubuntu)으로 SSH 접속하는 연습"** 이 가능해져요. 진짜 원격 서버에 접속하는 것과 거의 똑같은 경험이에요.

## 3. 실습 단계

### Step 1. Docker가 잘 깔려 있는지 확인

#### 🎯 목적
앞으로 모든 게 Docker 위에서 돌아가니까, 우선 Docker가 정상 동작하는지부터 확인.

#### ⌨️ 명령어
```bash
docker --version
```

#### 🔍 뜯어보기
- `docker` → Docker 명령어
- `--version` → "버전 알려줘" 라는 옵션

#### ✅ 성공 결과
```text
Docker version 27.x.x
```
이런 줄이 뜨면 OK. **버전 숫자가 보이기만 하면 통과**예요.

#### ⚠️ 실패하면?
- `command not found: docker` → Docker(또는 OrbStack)가 안 깔렸거나 안 켜져 있음. OrbStack 앱 먼저 실행하세요.

---

### Step 2. Ubuntu 컨테이너 만들고 들어가기

#### 🎯 목적
빈 Ubuntu 컨테이너 한 칸을 새로 만들고, 그 안으로 *들어가서* 명령어를 칠 수 있는 상태로 만든다.

#### ⌨️ 명령어
```bash
docker run -it --name ubuntu-ssh ubuntu:latest bash
```

#### 🔍 한 단어씩 뜯어보기
| 부분 | 뜻 |
|---|---|
| `docker run` | "컨테이너 새로 만들고 실행해줘" |
| `-it` | `-i` (입력 받을 수 있게) + `-t` (터미널 화면 줘) → **"내가 직접 키보드로 명령칠 수 있게 해줘"** |
| `--name ubuntu-ssh` | 컨테이너 이름을 `ubuntu-ssh` 로 지정 (안 정하면 랜덤 이름) |
| `ubuntu:latest` | "최신 Ubuntu 이미지로 만들어줘" (`:latest` = 최신 버전 태그) |
| `bash` | 컨테이너 들어가자마자 실행할 프로그램 = bash 셸 |

#### ✅ 성공 결과
```bash
root@a3f8d9e2c1b0:/#
```
**프롬프트(`#` 앞부분)가 바뀌었어요!** 이게 핵심이에요.
- `root` → 현재 로그인된 사용자 (최고 관리자)
- `a3f8d9e2c1b0` → 컨테이너 ID (랜덤 문자열, 사람마다 다름)
- `/` → 현재 위치 (Linux 루트 폴더)
- `#` → "너 지금 root다" 라는 표시 (일반 유저는 `$`)

이제부터 치는 명령어는 **Mac이 아니라 Ubuntu 안에서** 실행돼요.

#### 💡 컨테이너에서 나가고 다시 들어오기
- 나가기: `exit` 입력 → Mac으로 돌아옴
- 다시 들어가기: `docker start ubuntu-ssh && docker exec -it ubuntu-ssh bash`
  - `start` = 멈춘 컨테이너 다시 켜기
  - `exec -it ... bash` = 켜진 컨테이너 안으로 들어가서 bash 실행

---

### Step 3. 패키지 목록 업데이트

#### 🎯 목적
**프로그램 설치하기 전 워밍업**. Ubuntu의 "앱스토어 목록"을 최신으로 갱신.

#### ⌨️ 명령어
```bash
apt update
```

#### 🤔 이게 왜 필요해?
`apt`는 Ubuntu의 앱스토어 같은 거예요. 근데 컨테이너를 처음 만들면 **앱스토어 목록이 오래된 상태**예요. 마치 옛날 종이 카탈로그를 들고 가게 가는 느낌. 

`apt update`는 그 카탈로그를 **온라인 최신판으로 갱신**하는 거예요.

#### ⚠️ 헷갈리지 마세요
- `apt update` → 카탈로그(목록) 갱신만 함. **프로그램 자체는 업데이트 안 함**
- `apt upgrade` → 진짜로 프로그램들을 최신 버전으로 업그레이드 (이건 다른 명령어)

#### ✅ 성공 결과
주르륵 다운로드 받는 메시지가 나오고 마지막에:
```text
Reading package lists... Done
```

---

### Step 4. OpenSSH Server 설치

#### 🎯 목적
SSH 접속을 받아줄 *서버 프로그램*을 깐다.

#### ⌨️ 명령어
```bash
apt install openssh-server -y
```

#### 🔍 뜯어보기
- `apt install` → "설치해줘"
- `openssh-server` → 설치할 프로그램 이름
- `-y` → 중간에 "정말 깔까요? (y/n)" 같은 질문이 나와도 **자동으로 y(yes)** 누름

#### 🤔 SSH 서버가 정확히 뭐 하는 거야?
컨테이너 안에서 항상 켜져서 **"누구든 SSH로 들어오면 받아드릴게"** 하고 대기하는 직원이라고 생각하면 돼요.

비유로:
- 우리가 사이트(예: 네이버)에 접속할 때 → 네이버 컴퓨터엔 **웹 서버**가 돌고 있음
- 우리가 SSH로 어떤 컴퓨터에 접속할 때 → 그 컴퓨터엔 **SSH 서버**가 돌고 있어야 함

#### ✅ 성공 결과
설치 진행되고 마지막에 `done.` 같은 메시지.

---

### Step 5. vim 텍스트 편집기 설치

#### 🎯 목적
설정 파일을 수정하려면 텍스트 편집기가 필요한데, **Ubuntu 최소 이미지에는 편집기가 없을 수 있어요**. 그래서 깔아주는 단계.

#### ⌨️ 명령어
```bash
apt install vim -y
```

#### 🤔 vim 말고 다른 거 써도 돼?
네! `nano`가 훨씬 쉬워서 초보자에게 추천돼요:
```bash
apt install nano -y
```
하지만 **Linux 실무에선 vim이 표준** 같은 거라, 이번 기회에 익혀두면 좋아요.

> ⚠️ vim 처음 쓰면 *나가는 법을 몰라서 갇히는* 게 진짜 흔한 일이에요. 그래서 아래에 [vim 치트시트](#4-vim-치트시트-처음-쓰는-사람용) 따로 정리해뒀어요.

### Step 6. SSH 설정 파일 열기

#### 🎯 목적
SSH 서버의 동작 규칙이 적혀 있는 파일을 열어서 *수정*할 준비.

#### ⌨️ 명령어
```bash
vim /etc/ssh/sshd_config
```

#### 🔍 이 파일은 뭐야?
- `/etc/` → Linux에서 **설정 파일들이 모여 있는 폴더**. (etc = "etcetera"가 아니라 관습적으로 설정 폴더로 씀)
- `ssh/sshd_config` → SSH 서버(`sshd`)의 설정(`config`)
- **결론**: "SSH 서버가 어떤 규칙으로 동작할지 적힌 파일"

이 파일을 수정하면 SSH 서버의 동작이 바뀌어요.

### Step 7. SSH 포트 번호 바꾸기 (22 → 20022)

#### 🎯 목적
SSH가 받는 "문 번호"를 기본값(22)에서 다른 번호(20022)로 바꿔서 **보안 강화**.

#### 📝 수정 내용
파일에서 이 줄을 찾아서:
```text
#Port 22
```

이렇게 바꾸기:
```text
Port 20022
```

#### 🔍 변경된 두 가지
1. **`#` 제거** — `#`은 "이 줄은 주석이에요, 무시하세요" 라는 표시. 그래서 `#Port 22`는 사실 *적용 안 된 상태*. `#`을 지워야 진짜 적용됨.
2. **`22` → `20022`** — 포트 번호 변경.

#### 🤔 포트가 정확히 뭐야?
컴퓨터 한 대에는 IP 주소 하나만 있어요. 근데 그 컴퓨터가 동시에 여러 서비스(SSH, 웹, 이메일 등)를 제공할 수 있죠. 그럴 때 구분하는 게 **포트**예요.

비유:
- 아파트 단지 주소 = IP
- 동·호수 = 포트

자주 쓰는 포트:
| 포트 | 용도 |
|---|---|
| 22 | SSH |
| 80 | HTTP (웹) |
| 443 | HTTPS (보안 웹) |
| 3306 | MySQL |
| 5432 | PostgreSQL |
| 6379 | Redis |

#### 🤔 왜 굳이 22에서 다른 번호로 바꿔?
22는 **누구나 아는 SSH 기본 포트**예요. 인터넷에 떠도는 자동 해킹 봇들은 1초에도 수십만 번씩 전 세계의 22번 포트를 두드려요.

포트를 20022처럼 바꾸면:
- 🎯 **봇이 함부로 못 찾음** (브루트포스 공격 감소)
- 🛡️ **로그가 깨끗해짐** (불필요한 접속 시도 ↓)
- ⚠️ 단, 이건 **보안의 진짜 핵심은 아님**. 진짜 보안은 키 인증, 방화벽 등. 그래도 첫 단계로 좋아요.

#### 💡 왜 하필 20022야?
별 이유 없어요. 22를 기억하기 쉽게 앞에 200을 붙인 거예요. 1024~65535 사이 아무 번호나 가능 (1024 이하는 시스템 예약 포트).

### Step 8. root로 직접 SSH 접속 차단

#### 🎯 목적
"마스터키 가진 사람(root)이 SSH로 바로 들어오는 건 막자" → **보안 강화의 핵심**.

#### 📝 수정 내용
이 줄을 찾아서:
```text
#PermitRootLogin prohibit-password
```

이렇게 바꾸기:
```text
PermitRootLogin no
```

#### 🔍 의미
- `PermitRootLogin` = "root 로그인 허용?"
- `no` = "절대 안 됨"

#### 🤔 root가 SSH로 못 들어오면 어떻게 관리해?
**일반 사용자로 로그인 → 필요할 때만 `sudo` 로 잠깐 root 권한 빌려쓰기** 패턴이 표준이에요.

비유:
- ❌ 마스터키로 매번 다님 → 잃어버리면 끝장
- ✅ 자기 열쇠로 다니다가 필요할 때만 마스터키 빌려씀 → 마스터키 보호됨

#### 🤔 왜 이게 보안에 중요해?
공격자 입장에서 생각해보세요:
- root 직접 로그인 허용 → "username 알 필요 없음, root 비밀번호만 맞추면 끝"
- root 차단 → "username이 뭔지부터 알아내야 됨" → **공격 난이도 ↑**

### Step 9. /run/sshd 폴더 만들기

#### 🎯 목적
SSH 서버가 실행되면서 **임시로 쓰는 작업 공간**(폴더)을 미리 만들어둔다.

#### ⌨️ 명령어
```bash
mkdir -p /run/sshd
```

#### 🔍 뜯어보기
- `mkdir` = **m**a**k**e **dir**ectory = "폴더 만들어"
- `-p` = **p**arents = "중간 폴더도 같이 만들고, 이미 있으면 에러 내지 마"
- `/run/sshd` = 만들 폴더 경로

#### 🤔 -p 옵션이 왜 필요해?
예를 들어 `/run/sshd` 만들고 싶은데 `/run` 폴더 자체가 없다면? `-p` 없으면 에러가 나요. `-p` 붙이면 알아서 `/run`도 만들고 그 안에 `sshd`도 만들어줘요. **안전장치**라고 생각하면 됨.

#### 🤔 /run/sshd 가 정확히 뭐야?
SSH 서버가 돌아갈 때 만드는 *임시 파일들*이 들어가는 곳:
- PID 파일 (sshd 프로세스 번호)
- 소켓 파일 (통신 통로)
- 권한 분리용 임시 디렉토리

#### 🤔 보통은 자동으로 안 만들어져?
일반 Linux는 `systemd` 라는 시스템 관리자가 자동으로 만들어줘요. 근데 **우리가 쓰는 Docker Ubuntu 최소 이미지에는 systemd가 없어요**. 그래서 직접 만들어야 함.

#### ⚠️ 안 만들면 어떻게 돼?
sshd 실행하면 이런 에러:
```text
Missing privilege separation directory: /run/sshd
```

---

### Step 10. SSH 서버 실행

#### 🎯 목적
드디어 SSH 서버를 켠다!

#### ⌨️ 명령어
```bash
/usr/sbin/sshd
```

#### 🔍 뜯어보기
- `/usr/sbin/sshd` = sshd 프로그램의 **전체 경로**
- Linux에선 `sbin` 폴더에 시스템 관리용 프로그램들이 들어 있음 (s = system)

#### 🤔 그냥 `sshd` 라고 치면 안 돼?
환경 변수(PATH)에 따라 될 수도, 안 될 수도 있어요. 전체 경로(`/usr/sbin/sshd`)를 쓰면 **무조건 확실히 실행**돼서 안전해요.

#### ✅ 성공 결과
**아무것도 안 나오면 성공!** 🎉

처음엔 헷갈리는데, Linux 데몬은 **조용히 백그라운드로 도는 게 정상**이에요. 메시지가 없으면 "잘 켜졌다"는 뜻.

#### ⚠️ 메시지가 나오면?
에러 메시지가 나왔다면 뭔가 잘못된 거예요. 보통:
- `Missing privilege separation directory` → Step 9 안 했음
- `Address already in use` → 이미 sshd가 돌고 있음 (재시작 필요, Step 12 참고)

---

### Step 11. SSH가 진짜 켜졌는지 확인

#### 🎯 목적
"진짜 sshd가 돌고 있나?" 눈으로 확인.

#### ⌨️ 명령어
```bash
ps -ef | grep sshd
```

#### 🔍 뜯어보기
- `ps` = **p**rocess **s**tatus = "지금 돌아가는 프로그램 보여줘"
- `-ef` = 전체 프로세스 자세히 (e = every, f = full)
- `|` = **파이프** = "왼쪽 결과를 오른쪽에 넘겨줘"
- `grep sshd` = 결과에서 `sshd` 글자가 들어간 줄만 골라줘

**한 문장 해석**: "지금 돌아가는 모든 프로그램 중에 sshd 들어간 거 보여줘"

#### ✅ 성공 결과
```text
root   123   1   0   10:00   ?   00:00:00   /usr/sbin/sshd
root   456   xxx 0   10:05   ?   00:00:00   grep sshd
```

`/usr/sbin/sshd` 줄이 있으면 성공. (밑의 `grep sshd` 줄은 *지금 친 명령어 자체*라 무시)

#### 💡 파이프(`|`)는 Linux의 마법
"명령어 A 결과를 명령어 B에 넘긴다"는 개념인데, 이걸 알고 나면 Linux 명령어가 진짜 강력해져요. 예:
```bash
ps -ef | grep python | grep -v grep | wc -l   # 돌고 있는 python 프로세스 개수
```

---

### Step 12. 설정 바꿨으면 재시작

#### 🎯 목적
**아주 중요한 개념**: 설정 파일(`sshd_config`) 수정해도, *이미 돌고 있는* sshd는 옛날 설정을 기억해요. 새 설정 적용하려면 재시작 필수.

#### ⌨️ 명령어
```bash
pkill sshd          # 기존 sshd 다 죽이기
/usr/sbin/sshd      # 새로 시작
```

#### 🔍 뜯어보기
- `pkill` = **p**rocess **kill** = "이름이 ~인 프로세스 다 죽여"
- `pkill sshd` = sshd라는 이름 가진 프로세스 전부 종료

#### 🤔 왜 재시작이 필요해?
프로그램은 **시작할 때만 설정 파일을 읽어요**. 한 번 시작한 뒤로는 그 정보를 메모리에 가지고 일함. 그래서 설정을 바꾸면:

```text
설정 변경 ❌ 영향 없음 ← 기존 프로세스
설정 변경 ✅ 적용됨   ← 재시작 후 새 프로세스
```

#### 💡 좀 더 부드러운 방법
실무에선 `pkill` 대신 이렇게 많이 써요:
```bash
kill -HUP $(pgrep sshd)
```
- `HUP` 시그널은 "설정 다시 읽어!" 라는 의미라 진행 중 연결은 안 끊김. 다만 이번엔 컨테이너 최소 환경이라 단순하게 `pkill` 후 재실행 한 거예요.

---

### Step 13. 포트가 열렸는지 확인

#### 🎯 목적
"sshd가 진짜 20022 포트에서 손님 기다리고 있나?" 확인.

#### ⌨️ ss 명령어 없으면 설치
```bash
apt install iproute2 -y
```

#### ⌨️ 포트 상태 확인
```bash
ss -tlnp | grep ssh
```

#### 🔍 뜯어보기
- `ss` = **s**ocket **s**tatistics = 네트워크 연결 상태 보기 도구
- `-t` = TCP만
- `-l` = **l**isten 상태인 것만 (= 누가 들어오기를 기다리는 포트)
- `-n` = 숫자로 표시 (이름 변환 안 함, 빠름)
- `-p` = 어느 **p**rogram이 쓰는지도 보여줘
- `| grep ssh` = 그 중에 ssh 들어간 것만

#### ✅ 성공 결과
```text
LISTEN  0  128  0.0.0.0:20022  0.0.0.0:*  users:(("sshd",pid=...))
```

핵심은 **`0.0.0.0:20022`** 부분.
- `0.0.0.0` = "어디서 들어오는 연결이든 다 받음"
- `20022` = 우리가 설정한 포트
- → **"sshd가 20022 포트에서 손님 기다리는 중"** 이라는 뜻

#### 🤔 결과가 안 나오면?
- sshd가 안 켜진 상태일 수도 (Step 11에서 다시 확인)
- 설정이 적용 안 됐을 수도 (Step 12 재시작 다시)

---

### Step 14. 내가 누구로 로그인 중인지 확인

#### ⌨️ 명령어
```bash
whoami
```

#### ✅ 결과
```text
root
```

#### 🔍 의미
지금 컨테이너에서 **root** 사용자로 작업 중이라는 뜻. 모든 명령어를 최고 관리자 권한으로 실행할 수 있는 상태.

## 4. vim 치트시트 (처음 쓰는 사람용)

vim은 **모드가 있는** 편집기라서 처음엔 헷갈려요. "타자 치는 모드"와 "명령 모드"가 따로 있음.

### 🔑 가장 중요한 개념: 모드

```text
┌─────────────────┐
│  일반 모드      │ ← vim 켜자마자 여기. 키보드는 *명령용*
│  (Normal Mode)  │   (h,j,k,l로 이동, dd로 줄 삭제 등)
└────────┬────────┘
         │ i 누르면
         ▼
┌─────────────────┐
│  입력 모드      │ ← 진짜 글자 입력하는 모드
│  (Insert Mode)  │   화면 아래에 -- INSERT -- 표시됨
└────────┬────────┘
         │ ESC 누르면
         ▼
      일반 모드로 복귀
```

### 📝 진짜 자주 쓰는 명령어 (이것만 알아도 됨)

| 키 | 의미 | 언제 씀 |
|---|---|---|
| `i` | 입력 모드 진입 | 타이핑 시작할 때 |
| `ESC` | 일반 모드로 빠져나옴 | 입력 끝났을 때 |
| `:w` | 저장 | 변경사항 저장 (안 나감) |
| `:q` | 종료 | 변경 없을 때만 가능 |
| `:wq` | 저장+종료 | **이거 가장 많이 씀** |
| `:q!` | 저장 안 하고 강제 종료 | 실수했을 때 빠져나오기 |
| `/단어` | 단어 검색 | n으로 다음 결과 이동 |
| `dd` | 현재 줄 삭제 | (일반 모드에서) |
| `u` | 되돌리기(undo) | (일반 모드에서) |

### 🆘 vim에 갇혔을 때

처음 vim 켜고 *아무거나 누르다가 못 나가는* 게 진짜 흔해요. 무조건 이렇게:

```text
1. ESC를 3번 정도 누른다 (확실하게 일반 모드로)
2. :q! 입력
3. Enter
```

이러면 100% 빠져나옴.

### 🧭 sshd_config 수정 흐름 예시

```text
1. vim /etc/ssh/sshd_config       ← 파일 열기
2. /Port 입력 후 Enter            ← "Port" 단어 찾기
3. i 누르기                       ← 입력 모드 진입
4. #Port 22 → Port 20022로 수정
5. ESC                            ← 일반 모드로
6. :wq + Enter                    ← 저장하고 종료
```

## 5. 자주 만나는 에러와 해결법

### ❌ `Missing privilege separation directory: /run/sshd`
**원인**: `/run/sshd` 폴더가 없음
**해결**: 
```bash
mkdir -p /run/sshd
```

---

### ❌ `Could not load host key: /etc/ssh/ssh_host_rsa_key`
**원인**: SSH 호스트 키가 안 만들어져 있음
**해결**: 
```bash
ssh-keygen -A    # 모든 종류 호스트 키 자동 생성
```

---

### ❌ `Address already in use`
**원인**: 이미 sshd가 그 포트에서 돌고 있음
**해결**:
```bash
pkill sshd
/usr/sbin/sshd
```

---

### ❌ `apt: command not found`
**원인**: Ubuntu가 아닌 다른 Linux 배포판일 수 있음 (CentOS 등은 `yum`/`dnf` 사용)
**해결**: 컨테이너 이미지를 `ubuntu:latest`로 만들었는지 확인

---

### ❌ vim에서 못 나옴 / 명령이 안 먹힘
**해결**:
```text
ESC ESC ESC  →  :q!  →  Enter
```

---

### ❌ 설정 바꿨는데 적용 안 됨
**원인**: sshd 재시작 안 함
**해결**: Step 12 다시

---

### ❌ `ss: command not found`
**원인**: `iproute2` 패키지 없음
**해결**:
```bash
apt install iproute2 -y
```

## 6. 전체 흐름 한눈에 보기

```text
[ Mac 터미널 ]
     │
     │ docker --version              ← Docker 확인
     │ docker run -it --name ...     ← Ubuntu 컨테이너 생성+진입
     ▼
[ Ubuntu 컨테이너 안 ]
     │
     │ apt update                    ← 패키지 목록 갱신
     │ apt install openssh-server -y ← SSH 서버 설치
     │ apt install vim -y            ← 편집기 설치
     │
     │ vim /etc/ssh/sshd_config      ← 설정 파일 수정
     │   ├─ #Port 22  →  Port 20022
     │   └─ #PermitRootLogin ...  →  PermitRootLogin no
     │
     │ mkdir -p /run/sshd            ← sshd 작업 폴더 생성
     │ /usr/sbin/sshd                ← SSH 서버 시작
     │
     │ ps -ef | grep sshd            ← 켜졌나 확인
     │ pkill sshd && /usr/sbin/sshd  ← (설정 바꿨으니) 재시작
     │
     │ apt install iproute2 -y       ← ss 명령용
     │ ss -tlnp | grep ssh           ← 20022 포트 확인
     │
     │ whoami                        ← 현재 사용자 확인
     ▼
[ 완료! ]
```

## 📌 마무리: 이번에 진짜 배운 것

이 실습의 진짜 가치는 명령어 외우기가 아니에요. **이런 사고방식**을 배운 거예요:

1. **"서버는 그냥 프로그램이 돌고 있는 컴퓨터다"**  
   특별한 게 아니에요. sshd 라는 프로그램 하나 켜면 SSH 서버, nginx 켜면 웹서버.

2. **"설정 = 파일"**  
   Linux에선 거의 모든 설정이 텍스트 파일이에요. 그래서 vim/nano 같은 편집기가 중요한 거예요.

3. **"수정했으면 재시작"**  
   프로그램은 시작할 때 설정을 읽어요. 바꾸면 다시 시작해야 적용됨. (당연한 것 같지만 진짜 자주 까먹음)

4. **"보안은 기본값을 바꾸는 것부터"**  
   포트 변경, root 차단처럼 *기본값을 그대로 두지 않는 것*만으로도 보안이 한 단계 올라가요.

5. **"컨테이너 = 망가뜨려도 되는 연습장"**  
   Docker의 가장 큰 가치는 **부담 없이 실험할 수 있다**는 거예요. 망가지면 지우고 다시 만들기 = 5초.
